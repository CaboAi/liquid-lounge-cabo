## Gaps Summary

There is one root-cause blocker accounting for four of the five failures: **no live Supabase project is provisioned**. The target project (pamzfhiiuvmtlwwvufut) has a non-existent DNS record. All five migration files (001 through 005) are syntactically correct and structurally complete, but `supabase db push` has never succeeded. Until migrations are applied:

- The 8 canonical tables do not exist in any live database
- RLS policies are not enforced anywhere
- The `handle_new_user` trigger is not installed — sign-ups create auth.users rows but no profiles rows
- The 3 seed plan rows are not in any plans table
- The 5 performance indexes are not created
- `lib/supabase/types.ts` cannot be generated from a live schema

The auth code layer (sign-in form, signout route, reset-password page, middleware, unit tests) is fully implemented and all 17 unit tests pass. The implementation is correct. The gap is exclusively in the database infrastructure layer.

**To close all DB gaps in one step:**
1. Provision a new Supabase project at https://supabase.com
2. Update `.env.local` with the new project's `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY`
3. Run `npx supabase link --project-ref <new-project-ref>`
4. Run `npx supabase db push`
5. Run `npx supabase gen types typescript --project-id <new-project-ref> > lib/supabase/types.ts`
6. Verify human verification items 1–4 above
