Create supabase/migrations/005_trigger.sql with this exact content (do not alter
the SQL — it matches the verified pattern from research):

  -- Auto-profile-creation trigger
  -- Source: Supabase official docs + supabase/migrations/auto_profile_creation.sql reference
  -- SECURITY DEFINER is required: allows the trigger to write to public.profiles
  -- even when RLS is enabled and auth.uid() is null during trigger execution.

  CREATE OR REPLACE FUNCTION handle_new_user()
  RETURNS TRIGGER
  LANGUAGE plpgsql
  SECURITY DEFINER SET search_path = public
  AS $$
  BEGIN
    INSERT INTO public.profiles (id, full_name, role, credits)
    VALUES (
      NEW.id,
      COALESCE(NEW.raw_user_meta_data ->> 'full_name', ''),
      'member',
      0
    )
    ON CONFLICT (id) DO NOTHING;
    RETURN NEW;
  END;
  $$;

  DROP TRIGGER IF EXISTS on_auth_user_created ON auth.users;
  CREATE TRIGGER on_auth_user_created
    AFTER INSERT ON auth.users
    FOR EACH ROW EXECUTE PROCEDURE handle_new_user();

Then apply:

Verify in Supabase Dashboard: Database → Functions → should show handle_new_user.
Database → Triggers → auth.users → should show on_auth_user_created.
